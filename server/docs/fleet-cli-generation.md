# Fleet CLI Generation

This document describes the current architecture for the protobuf-driven
`fleetcli` CLI.

The Fleet CLI uses the repo's existing protobuf sources and generated Go
packages. Generated command source files are written under `server/cmd/fleetcli/`,
while a small handwritten runtime provides shared transport, auth, and helper
logic.

## Pipeline

The Fleet generation flow is:

```text
proto/
  └─[buf build --as-file-descriptor-set]──────────────┐
                                                      │
      .cache/fleet-cli/fleet-descriptor-set.bin       │
                                                      │
.cache/fleet-cli/fleet-descriptor-set.bin
+ server/tools/generate-fleet-cli/overrides.json
  └─[server/tools/generate-fleet-cli]─────────────────┐
                                                      │
      server/cmd/fleetcli/cmd_<group>.go              │
      server/cmd/fleetcli/cmd_commands.go             │
      .cache/fleet-cli/fleet-cli-report.json          │
```

`just gen-fleet-cli` builds the descriptor set from the local `proto/` module
and then runs the generator. `just gen` also runs this after protobuf
generation.

## Handwritten vs Generated

### Handwritten Fleet runtime

These files stay handwritten:

- `server/cmd/fleetcli/main.go`
- `server/cmd/fleetcli/client.go`
- `server/cmd/fleetcli/manual.go`
- `server/cmd/fleetcli/generated_runtime.go`

The handwritten layer owns:

- global CLI flags
- HTTP JSON RPC transport and auth dispatch
- session login and API key management commands
- the telemetry-backed `performance` command
- shared helpers used by generated commands

### Generated Fleet command layer

These files are generated by `server/tools/generate-fleet-cli`:

- `server/cmd/fleetcli/cmd_<group>.go`
- `server/cmd/fleetcli/cmd_commands.go`

The generated files define command groups and subcommands, but delegate request
construction and execution to helpers in `server/cmd/fleetcli/generated_runtime.go`.

## Generator Inputs

### Input 1: protobuf files

The generator reads a descriptor set produced from the repo's canonical
`proto/` module:

```bash
buf build -o .cache/fleet-cli/fleet-descriptor-set.bin --as-file-descriptor-set
```

It imports the existing generated Go types from
`github.com/block/proto-fleet/server/generated/grpc/...`. It does not create or
use a duplicate protobuf Go tree.

### Input 2: Fleet CLI overrides

`server/tools/generate-fleet-cli/overrides.json` supplies the CLI behavior that
descriptors do not encode cleanly, including:

- generated service selection via `all_methods`
- custom group names such as `miners`
- custom subcommand names such as `set-power-target`
- auth mode overrides: `anonymous`, `bearer`, `session`
- command aliases that split one service across multiple CLI groups, such as
  `groups` and `racks`
- fixed request fields and ignored request fields for generated commands
- `json_only` overrides for commands that should be request-file driven

## Generated Command Rules

### Grouping and naming

By default, the generator:

- strips a trailing `Service` suffix from the protobuf service name
- lowercases the result for the command group
- derives subcommand names from the RPC method verb and noun

Examples:

- `NetworkInfoService` -> `networkinfo`
- `ListSchedules` -> `list`
- `PauseSchedule` -> `pause`

Overrides are used when the default CLI name would be awkward or misleading.

### Request handling

For unary RPCs, the generator inspects the request and response descriptors and
chooses one of three output modes:

- `generated`
  simple scalar and enum fields become CLI flags
- `generated_json_fallback`
  simple fields become flags, and `--json` is also accepted for complex request
  fields
- `generated_json_only`
  the command requires `--json` because the request shape is intentionally or
  effectively too complex for generated flags

Responses containing `bytes` fields are not generated today because they
require custom binary or file handling.

### Auth handling

Generated commands do not implement auth logic directly. Instead, each command
is emitted with one auth mode:

- `anonymous`
- `bearer`
- `session`

The runtime helper chooses the correct client call path:

- `client.CallAnonymous(...)`
- `client.CallBearer(...)`
- `client.CallSession(...)`

## Special-Case Runtime Helpers

Most generated commands are thin wrappers, but the runtime includes a few
shared helpers for request shapes that need more than plain top-level flag
assignment.

### Miner command device selectors

`minercommand.v1.DeviceSelector` is handled specially in
`server/cmd/fleetcli/generated_runtime.go`.

Generated `minercommand` commands support:

- `--all-devices`
- `--device`
- `--group-id`
- `--group`
- `--rack-id`
- `--rack`

`--group` and `--rack` resolve exact collection labels through
`collection.v1.DeviceCollectionService/ListCollections`, then expand those
collections to device identifiers through `ListCollectionMembers`.

If a label does not exist, the CLI returns an error. If multiple collections
share the same label, the CLI returns an error and instructs the user to use
`--group-id` or `--rack-id`.

## Current Command Split

The generated command registry currently contributes:

- `groups`
- `minercommand`
- `miners`
- `networkinfo`
- `onboarding`
- `pools`
- `racks`
- `schedule`

The handwritten command registry currently contributes:

- `auth`
- `apikey`
- `performance`

`performance` remains handwritten because it is a telemetry-backed UX command
rather than a direct one-to-one generated wrapper over a single Fleet RPC.

## Coverage Report

Each `generate-fleet-cli` run writes `.cache/fleet-cli/fleet-cli-report.json`.

The report includes a summary plus one entry per protobuf RPC method. The
generator can currently emit these statuses:

- `generated`
- `generated_json_fallback`
- `generated_json_only`
- `deferred_streaming`
- `deferred_binary_response`
- `deferred_service_skipped`
- `deferred_method_skipped`
- `deferred_unselected`

This makes the generator boundary explicit and helps track what is covered,
what is intentionally deferred, and what still depends on handwritten CLI code
or future overrides.

## Development Workflow

Typical Fleet CLI regeneration flow:

1. Update protobufs in `proto/` or CLI behavior in `server/tools/generate-fleet-cli/overrides.json`
1. Run `just gen-fleet-cli`
1. Build with `just build-fleet-cli`
1. Inspect `.cache/fleet-cli/fleet-cli-report.json` for unexpected status changes
1. Smoke-test `fleetcli --help` and any touched command groups

`just gen` regenerates the Fleet CLI alongside the other generated code.

## Verification Guidance

For incremental verification:

1. `just gen-fleet-cli`
1. `just build-fleet-cli`
1. `./server/.fleet/fleetcli --help`
1. Inspect `.cache/fleet-cli/fleet-cli-report.json`

For live validation, prefer read-only RPCs first. In practice, safe checks have
included commands such as:

- `fleetcli miners list`
- `fleetcli networkinfo get`
- `fleetcli performance get`
- `fleetcli minercommand check-capabilities`
